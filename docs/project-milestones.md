## 一、方法论：如何划分里程碑

### 1.1 划分原则

| 原则 | 说明 |
|------|------|
| **依赖优先** | 被其他模块依赖的基础模块先交付（如品类管理是资源/范本的前置依赖） |
| **P0 优先** | 每轮迭代优先完成 P0 需求，确保核心链路可用 |
| **用户价值** | 每个里程碑交付后，用户可独立完成一个完整的业务子流程 |
| **风险前置** | 技术风险高的模块（如 WPS 编辑器集成）提前验证，不放在最后 |
| **可验收** | 每个里程碑结束时，有明确的验收标准和可演示的功能 |

### 1.2 优先级定义

| 优先级 | 定义 | 占比 |
|--------|------|------|
| **P0** | 核心功能缺失则系统不可用 | 12 / 156 ≈ 8% |
| **P1** | 主要功能，缺失则体验显著下降 | 120 / 156 ≈ 77% |
| **P2** | 增强功能，可后续迭代 | 17 / 156 ≈ 11% |
| **P3** | 锦上添花，可排期或延后 | 7 / 156 ≈ 4% |

### 1.3 模块依赖关系

```
品类管理（CM）          ← 最底层依赖
    ↓
资源管理（RM）          ← 依赖品类数据
    ↓
范本变量管理（GV）       ← 独立，但与范本强相关
    ↓
范本管理（TP）          ← 依赖品类 + 资源 + 变量
    ↓
在线编辑（WE）          ← 依赖范本
    ↓
招标文件管理（BD）       ← 依赖已发布范本
```
