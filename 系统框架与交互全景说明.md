# 招标文件范本系统：框架与交互全景说明

## 1. 文档目的

本文档用于统一前后端对当前系统的认知，明确：
- 业务模块边界与职责
- 页面交互与跨模块数据流转
- 前后端接口契约重点
- OnlyOffice 集成链路与回调机制

适用读者：前端开发、后端开发、联调与测试人员。

---

## 2. 系统定位与当前实现形态

系统是一个招标文件编制平台，覆盖从资源管理到范本编制、再到招标文件生成的完整链路。

当前形态：
- 前端：Next.js App Router + TypeScript + Tailwind + Lucide
- 数据：前端本地持久化（`localStorage`）+ 部分文档文件落盘（`public/documents`）
- 文档编辑：OnlyOffice Document Server（Docker）在线编辑
- 接口：Next.js API Routes（同仓 API 层）

---

## 3. 业务域模型（统一语言）

核心实体关系：
- `Industry`（产业领域）
  - `ProcurementCategory`（采购分类）
    - `Category`（品类）
      - `Template`（范本）
        - `TemplateSection`（范本章节）
        - `TemplateVariable`（变量定义）
      - `BidDocument`（招标文件）
        - `VariableValue`（变量取值）
- `TextFragment`（资源条款，含文本/资格条件/评标办法/合同条款四类模块化复用）

关键约束：
- 品类是范本创建的上游维度
- 范本是招标文件生成的模板来源
- 文本资源可被范本章节引用，文本更新会影响关联范本

---

## 4. 信息架构（页面与职责）

导航结构：
- 数据看板
- 品类管理
- 范本管理
- 资源管理（折叠组）
  - 文本管理
  - 资格条件管理
  - 评标办法管理
  - 合同条款管理
- 招标文件管理

各模块职责：
- 数据看板：展示系统运行态统计（已改为基于真实关联数据动态计算）
- 品类管理：维护三级树（产业领域→采购分类→品类）
- 资源管理：维护可复用条款，供范本侧引用/拷贝
- 范本管理：创建、筛选、导入、OnlyOffice 编辑、发布、导出
- 招标文件管理：基于范本生成项目文档并填充变量，推进草稿到定稿

---

## 5. 核心交互链路（端到端）

### 5.1 主业务流

1) 品类管理维护结构化分类  
2) 资源管理维护条款内容（文本/资格/评标/合同）  
3) 范本管理按品类创建范本并进入 OnlyOffice 编辑  
4) 招标文件管理基于已发布范本创建项目文档并填写变量  
5) 看板自动读取关联数据并形成统计与待办

### 5.2 资源到范本的关联

- 范本章节支持绑定 `textFragmentId`
- 文本更新后，通过章节引用关系可识别受影响范本
- 看板“文本更新影响”即基于“文本更新时间 > 范本更新时间 + 章节引用关系”计算

### 5.3 范本到招标文件的关联

- 招标文件创建时选择范本（优先使用范本模块的真实共享数据）
- 变量列表来自范本 `TemplateVariable[]`
- 招标文件保存后持久化变量值，计算完成率，并支持定稿状态推进

## 6. 数据持久化与本地存储设计

当前采用统一本地存储封装（`src/lib/mockData.ts`），已支持以下数据的读写持久化：

- `oo-industries`：品类树
- `oo-text-fragments`：文本资源
- `oo-templates`：范本
- `oo-bid-documents`：招标文件

策略说明：
- 首次读取：优先从 `localStorage` 恢复；无数据时回落默认初始数据
- 业务写入：页面状态更新后同步调用统一 setter 写回本地
- 结果：刷新页面后数据不丢失，模块间数据可流转

## 7. OnlyOffice 集成架构（重点）

### 7.1 关键接口

- `GET /api/documents/[id]`
  - 获取文档二进制（docx）
- `POST /api/documents/[id]`
  - 以 base64 上传/覆盖文档
- `GET /api/documents/[id]/editor-config`
  - 返回 OnlyOffice 初始化配置（含 token / callbackUrl）
- `POST /api/documents/[id]/callback`
  - 接收 OnlyOffice 回调并落盘
- `PUT /api/documents/[id]/heading`
  - 兜底创建文档标题（失败回退链路）

### 7.2 编辑链路

1) 前端打开范本编辑页  
2) 确保远端文档存在：导入 docx 时优先覆盖上传，非导入场景按章节生成  
3) 拉取 editor-config，加载 OnlyOffice API  
4) 初始化 `DocsAPI.DocEditor`  
5) OnlyOffice 自动保存并通过 callback 回传，服务端落盘  
6) 前端“保存并返回”更新范本元数据（更新时间等）

### 7.3 导出策略

- 优先下载 OnlyOffice 原生 docx
- 若判定为空白回退，则使用 HTML->DOCX 兜底导出

## 8. 数据看板计算口径（已去除演示数据）

已改为实时计算，不再写死展示值：

- 品类总数：汇总所有 `Category` 数量
- 范本总数：`Template` 总量，同时展示发布/草稿拆分
- 招标文件：`BidDocument` 总量，同时展示定稿/草稿拆分
- 待完成范本编制：`draft && editProgress < 100`
- 待审核范本：`draft && editProgress >= 100`
- 文本更新影响：章节引用文本中存在 `text.updatedAt > template.updatedAt`

## 9. 前后端接口契约建议（下一步服务化）

为后续从本地存储迁移到后端服务，建议统一如下接口域：

### 9.1 目录与资源域
- `GET /industries/tree`
- `POST /industries` / `PUT /industries/:id` / `DELETE /industries/:id`
- `GET /text-fragments`
- `POST /text-fragments` / `PUT /text-fragments/:id` / `DELETE /text-fragments/:id`

- `PUT /templates/:id`（元数据更新）
- `POST /templates/:id/publish`（状态流转）

- `GET /bid-documents`
- `POST /bid-documents`
- `PUT /bid-documents/:id/variables`

### 9.4 统计域
- `GET /dashboard/metrics`
- 返回卡片统计 + 待办聚合（便于前端一次请求渲染）

### 9.5 文档编辑域（OnlyOffice）
- 维持当前 API 结构可平滑迁移
- 补充签名校验、幂等处理、回调审计日志

## 10. 状态机与一致性要点

### 10.1 范本状态
- `draft` -> `published` -> `archived`
- 推荐仅允许单向流转，归档回退需管理员操作

### 10.2 招标文件状态
- `draft` -> `completed` -> `archived`
- 进入 `completed` 前应校验必填变量完整性

### 10.3 一致性建议
- 文本更新后：触发受影响范本标记（dirty/needs_review）
- 范本发布时：记录资源版本快照（便于审计）
- 招标文件生成时：记录范本版本号，避免后续变更影响已生成文档

## 11. 运行与配置要点

关键环境变量（OnlyOffice 联调）：
- `NEXT_PUBLIC_ONLYOFFICE_DOCUMENT_SERVER_URL`
- `ONLYOFFICE_INTERNAL_BASE_URL`
- `ONLYOFFICE_PUBLIC_BASE_URL`
- `ONLYOFFICE_CALLBACK_BASE_URL`
- `ONLYOFFICE_JWT_SECRET`

建议：
- 开发环境与演示环境分离
- 回调地址与文档访问地址统一可达性策略（Docker 内外网）
- 字体与导出一致性作为环境基线配置项

## 12. 联调检查清单（前后端共用）

- 菜单切换、筛选、状态流转是否实时生效
- 本地刷新后数据是否保持一致
- 文本更新后看板“影响数”是否变化
- 范本编辑回调是否落盘成功（非空 docx）
- 原生导出与兜底导出均可用
- 导入 docx 后内容、字体、样式、分页是否可接受
